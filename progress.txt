Bug encontrado y resuelto durante el Test 2 (acceso desde LAN):
  - Sintoma: desde Quest, celular y otra PC en la misma wifi, el backend
    devolvia pagina en blanco / "inalcanzable". Desde localhost en la PC
    host funcionaba perfecto. Firewall ya estaba abierto y Docker bindeaba
    0.0.0.0:8080 (verificado con netstat).
  - Causa: en backend/.env teniamos DOMAIN=localhost. El Caddyfile usa
    `{$SCHEME}{$DOMAIN:localhost}:{$PORT:8080}` como direccion del sitio,
    asi que Caddy solo aceptaba requests cuyo header Host fuera
    "localhost:8080". Cualquier request de la LAN llega con Host=<IP>:8080
    y Caddy no matcheaba ningun sitio -> respuesta vacia.
  - Fix: dejar DOMAIN= vacio en .env (modo local). Caddy interpreta
    http://:8080 como "escuchar en todos los hostnames". En prod volver
    a setear DOMAIN=api.tu-dominio.com (necesario para que Caddy emita
    el cert TLS especifico para ese host).
  - Comando para aplicar: docker compose up -d --force-recreate caddy
    (solo `up -d` no recrea si no cambio el compose; force-recreate si).

Bug encadenado descubierto enseguida (Test 2, segunda pasada):
  - Sintoma: tras cambiar DOMAIN= en .env, Caddy seguia devolviendo
    Content-Length: 0 a peticiones desde la IP de LAN.
  - Diagnostico via `docker exec backend-caddy-1 caddy adapt`: la config
    JSON adaptada mostraba "match":[{"host":["localhost"]}], es decir,
    Caddy seguia matcheando solo Host=localhost.
  - Causa real: en docker-compose.yml la linea era
        DOMAIN: ${DOMAIN:-localhost}
    La sintaxis `${VAR:-default}` (con dos puntos) cae al default si la
    var esta UNSET *o* VACIA. Por eso DOMAIN= se ignoraba y volvia a
    localhost antes de llegar al container.
  - Fix: cambiar a `${DOMAIN-localhost}` (sin dos puntos): solo cae al
    default si la var esta unset; acepta string vacio como valor valido.
  - Bug paralelo: BACKEND_URL en autoloads/data_manager.gd estaba en
    "http://localhost:8080". En Quest standalone, localhost = el Quest,
    no la PC. RESULT_CANT_CONNECT (2) en HTTPRequest.
    Fix temporal: hardcodear "http://192.168.2.12:8080".
    Pendiente: hacer BACKEND_URL configurable en runtime (user://
    backend_url.txt o ProjectSettings.get_setting con override) para no
    reexportar el APK cada vez que cambia la IP de LAN.
